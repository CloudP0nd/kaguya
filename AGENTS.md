# AGENTS.md — Kaguya エージェント行動規約

> このプロジェクトは日本語を標準言語とする。コミットメッセージ、ドキュメント、コメント、PR本文はすべて日本語で記述すること。
> ただし、コード内の識別子（変数名・関数名・クラス名）と技術用語は英語のままでよい。

---

## 1. プロジェクトの立ち位置

Kaguyaは **llama.cppを超えるCPU推論エンジン** である。

- **主戦力**: AVX-512 BF16 + VNNI（実証済み・KVM環境で動作確認）
- **ベアメタル専用**: AMX TILE/BF16/INT8（仮想環境では動かない）
- **互換性**: GGUFフォーマット互換 → llama.cppのモデル資産をそのまま利用可能
- **目標**: llama.cpp比 1.5x〜2x の推論スループット

「llama.cppと同じことをする」のではなく、「llama.cppにない最適化をする」ことが存在意義である。
既存のアプローチを再実装するだけで満足してはならない。

## 2. 開発フローとGit規約

### ブランチ戦略

| 種別 | 命名規則 | 例 |
|------|----------|-----|
| 機能追加 | `feat/phaseN-description` | `feat/phase3-gemm-kernels` |
| リファクタリング | `refactor/description` | `refactor/project-restructure` |
| バグ修正 | `fix/description` | `fix/sampler-priority-queue` |

### PRパターン

- **fork → upstream のクロス組織PR**
  - fork: `CloudP0nd/kaguya`
  - upstream: `Carvlly/kaguya`
  - ブランチをforkにpushし、upstreamに対してPRを作成する
- GitHub認証トークンは **AGENTS.mdには記述しない** — セッションごとにユーザーが提供する
- `gh` CLIが利用可能なら使用する。未導入の場合はGitHub APIを直接呼ぶ（Python urllib等）

### コミットメッセージ

- プレフィックス必須: `feat:` / `refactor:` / `fix:` / `docs:` / `test:`
- 日本語で記述する（例: `feat: AVX-512 BF16 GEMMカーネルの実装`）

### ドキュメント更新

- 各フェーズ完了時、**必ず** PLAN.mdの該当フェーズに完了マークを付ける
- 調査経緯・重要な発見は **ROAD.md** に記録する（PLAN.mdには書かない）
- 新たな「罠」を発見したら **AGENTS.mdの技術的制約セクション** に追記する

## 3. 技術的制約と「罠」ログ

過去に踏んだ地雷を再踏しないための必読情報。開発前に必ず一読すること。

### 3.1 命令セット関連

| 罠 | 詳細 | 対策 |
|---|---|---|
| **AMXはKVMで動かない** | CPUIDはAMX対応を報告するが、`LDTILECFG`がSIGSEGV。CPUID Leaf 0x1Dの返り値も不正。ハイパーバイザがAMXを正しくパススルーしていない | KVM環境ではAMXカーネルをスキップ。ベアメタルでのみ有効化 |
| **GCC 14は`tmm`レジスタを認識しない** | inline asmのclobber listに`tmm0`〜`tmm7`を書くとコンパイルエラー | clobberから`tmm`を削除。代わりに`"memory"`で全メモリclobberを指定 |
| **AVX-512 BF16/VNNIは実動作確認済** | `_mm512_dpbf16_ps`, `_mm512_dpbusd_epi32` がKVM環境でも正常動作 | これが主戦力。安心して使える |
| **KVMでFMA CPUIDビットが0になる** | CPUID Leaf 7 EBX[12]（FMA）がKVM環境で0を返す。しかしAVX-512が利用可能ならFMA/AVX2は必ず存在する | `cpu_features.cpp`でAVX-512検出時にFMA/AVX2を論理的に有効化する推論を追加済み |
| **GCCに`_mm512_exp_ps`が存在しない** | AVX-512にはexp()のintrinsicが無い | `exp(x) = 2^(x/ln2)` + 6次Taylor展開で独自実装。~1e-7精度を確認済み |

### 3.2 フォールバックチェーン

```
AMX (ベアメタル専用)
  → AVX-512 BF16/VNNI (主戦力・実証済み)
    → AVX2 + FMA
      → Scalar
```

カーネルディスパッチャはこの順序で最適カーネルを選択すること。
実行時にCPUID + XCR0で判定し、利用可能な最上位カーネルを使用する。

### 3.3 ビルド・コンパイル関連

| 罠 | 詳細 | 対策 |
|---|---|---|
| **numa.hが無い環境がある** | `<numa.h>`がインストールされていないとコンパイルエラー | `#ifdef KAGUYA_NUMA` でガード。CMakeで`find_path`して定義 |
| **`std::greater`のテンプレート引数** | `priority_queue`の比較に`std::greater<>{}`だと型推論が失敗 | `std::greater<ElementType>{}` のように明示的に型を指定 |

### 3.4 Git・GitHub関連

| 罠 | 詳細 | 対策 |
|---|---|---|
| **push先の不一致** | originがfork (`CloudP0nd`) を指している場合とupstream (`Carvlly`) を指している場合がある | push前に`git remote -v`で確認。PRはfork→upstreamの方向で作成 |
| **unrelated histories** | 異なるgit履歴同士のマージでエラー | `--allow-unrelated-histories` で対応（通常は発生しないはず） |

## 4. ビルド・テスト規約

### ビルド手順

```bash
mkdir -p build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j$(nproc)
```

### テスト実行

```bash
cd build && ./kaguya_tests
```

- **全テスト通過がコミットの前提** — テストが一つでも落ちる状態でコミットしてはならない
- 新機能追加時は必ずテストを書く — テストなきコミットは認めない

### 環境

- **C++23 + GCC 14.2.0** — 新機能を使ってよいが、GCC 14でのコンパイル可否を必ず確認
- **外部依存**: Google Test / Google Benchmark はCMake FetchContentで自動取得
- **NUMA**: `libnuma-dev`がインストールされていなくてもビルド可能（`KAGUYA_NUMA`が未定義になるだけ）

## 5. ガードレール

以下のルールに違反する実装は行わないこと。

### 絶対遵守

1. **PLAN.mdにない機能を実装しない** — 計画にない機能を追加したい場合は、まずPLAN.mdを更新しユーザーの承認を得ること
2. **テストが通らない状態でコミットしない** — WIPコミットでもテストは通すこと
3. **PLAN.mdが唯一の計画書** — 他のplanファイルが現れてもPLAN.mdを正とする。新たな計画書ファイルを作成しない
4. **既存のテストを削除・スキップしない** — テストが邪魔なら直すのであって、消すのではない
5. **認証情報をファイルに書き込まない** — トークン・パスワードはAGENTS.mdにもソースコードにも書かない

### 強く推奨

6. **1コミット = 1論理的変更** — リファクタリングと機能追加を混ぜない
7. **PR本文にテスト結果を記載する** — テスト数・通過数を明記
8. **新しい「罠」を発見したらAGENTS.mdを更新する** — 次のエージェントのために必ず残す
9. **大きな設計変更前にはPLAN.mdを更新する** — 実装が計画から逸脱したら先に計画を直す

## 7. ドキュメント体系

| ファイル | 役割 | 更新タイミング |
|----------|------|----------------|
| **PLAN.md** | 開発計画・アーキテクチャ設計・フェーズ定義 | フェーズ完了時・設計変更時 |
| **ROAD.md** | 開発経緯・調査記録・重要な発見 | フェーズ完了時・重要な技術的発見時 |
| **AGENTS.md** | エージェント行動規約・技術的制約・ガードレール | 新たな「罠」発見時・フロー変更時 |

PLAN.mdには「何を作るか」を書き、ROAD.mdには「どう進んだか・何に気づいたか」を書く。
この分離により、PLAN.mdを将来の開発の参照元として常にクリーンに保つ。
